# 架构设计文档

## 1. 架构总览

Harness Engineering 采用 **分层管控 + 事件驱动** 的架构模式，核心设计思想：

```
┌─────────────────────────────────────────────────────┐
│                   用户 / 管理层                       │
│            (人高位统筹，制度设计与验收)                  │
└──────────────────────┬──────────────────────────────┘
                       │ 任务下发 / 制度定义
                       ▼
┌─────────────────────────────────────────────────────┐
│              Workflow 指挥链条层                       │
│     (全链路分级调度，Agent 间衔接、流转、回退)           │
└──────────────────────┬──────────────────────────────┘
                       │ 任务路由
                       ▼
┌─────────────────────────────────────────────────────┐
│              Agent 分工执行层                          │
│  PM调度 → 需求拆解 → 方案设计 → 风控 → 开发 → 审查 → 测试│
└──────────────────────┬──────────────────────────────┘
                       │ 约束 & 验收
          ┌────────────┼────────────┐
          ▼            ▼            ▼
┌──────────────┐ ┌──────────┐ ┌──────────────┐
│  Rule 底线层  │ │ Skill 库 │ │ Scripts 门禁 │
│ (刚性约束)    │ │ (标准流程)│ │ (自动验收)    │
└──────────────┘ └──────────┘ └──────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│              MCP 外接层                               │
│     Git / CI-CD / 看板 / 工单 / 监控 / 数据库          │
└─────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│              Assets 沉淀层                            │
│     知识库 / 错题集 / 经验库 / 模板库 / dev-map        │
└─────────────────────────────────────────────────────┘
```

## 2. 数据流

```
用户需求
   │
   ▼
[PM调度Agent] ──路由──▶ [需求拆解Agent]
                            │
                            ▼ STRUCTURED_SPEC
                     [方案设计Agent]
                            │
                            ▼ DESIGN_DOC
                     [闸门风控Agent] ──不通过──▶ 回退
                            │ 通过
                            ▼
                     [开发执行Agent]
                            │
                            ▼ CODE_OUTPUT
                     [代码审查Agent] ──不通过──▶ 回退
                            │ 通过
                            ▼
                     [测试验收Agent] ──不通过──▶ 回退
                            │ 通过
                            ▼
                       交付完成 ✅
```

每个 Agent 间流转时，必须经过 **Scripts 门禁校验**，达标方可推进到下一环节。

## 3. Agent 交互协议

### 3.1 统一消息格式

```yaml
message:
  id: "msg-uuid"
  from: "requirement-analyzer"
  to: "solution-designer"
  type: "handover"       # handover | rollback | query | notify
  priority: "normal"     # urgent | high | normal | low
  payload:
    task_id: "task-001"
    spec_doc: "path/to/spec.md"
    context: {}
  timestamp: "2026-04-23T20:00:00Z"
```

### 3.2 交接规则

- **handover**: 前序 Agent 完成任务，移交下一环节，必须附带产出物路径
- **rollback**: 当前环节不通过，回退到指定 Agent，必须附带原因
- **query**: 向其他 Agent 请求信息，不涉及任务转移
- **notify**: 通知其他 Agent 状态变更，无需响应

### 3.3 回退策略

```
回退层级:
  L1: 回退到上一个 Agent（小修小补）
  L2: 回退到方案设计阶段（方案调整）
  L3: 回退到需求拆解阶段（需求重新理解）
  L4: 回退到 PM 调度重新路由（任务重新定义）
```

## 4. Rule 约束体系

### 4.1 约束层级

```
L0 - 法律合规红线（不可逾越，违反即终止）
  └── 数据隐私 / 安全漏洞 / 知识产权
L1 - 工程规范红线（强制执行，违反即回退）
  └── 代码风格 / 架构规范 / 测试覆盖
L2 - 团队最佳实践（推荐执行，偏离需说明）
  └── 设计模式 / 性能优化 / 可维护性
L3 - 项目特定约定（按需执行，灵活调整）
  └── 命名约定 / 目录结构 / 工具选择
```

### 4.2 约束执行机制

- 每个 Agent 在执行前必须读取相关 Rule
- Scripts 在每个节点自动校验 Rule 合规性
- 违反 L0/L1 的产出直接拦截，不进入下一环节
- 违反 L2/L3 的产出标记警告，由 PM Agent 决策

## 5. Skill 标准化体系

### 5.1 Skill 定义规范

每个 Skill 必须包含：

```yaml
skill:
  name: "skill-name"
  version: "1.0.0"
  description: "简明描述"
  trigger: "触发条件"
  inputs: []           # 输入参数定义
  outputs: []          # 输出产物定义
  steps: []            # 标准执行步骤
  rules: []            # 关联 Rule
  scripts: []          # 关联验收脚本
  templates: []        # 关联模板
```

### 5.2 Skill 调用链

```
Agent 接收任务
   │
   ├─ 匹配 Skill
   │    └─ 按 trigger 条件自动匹配
   │
   ├─ 加载 Rule 约束
   │    └─ 读取关联的 Rule 红线
   │
   ├─ 执行 Skill Steps
   │    └─ 严格按步骤执行
   │
   └─ 运行 Scripts 验收
        └─ 自动校验产出物
```

## 6. 门禁验收体系

### 6.1 门禁节点

| 节点 | 门禁脚本 | 校验内容 |
|------|---------|---------|
| 需求 → 设计 | `check-spec` | SPEC 完整性、可量化性、无歧义性 |
| 设计 → 风控 | `check-design` | 方案合规性、架构合理性、风险标注 |
| 风控 → 开发 | `check-risk` | 技术可行性、资源匹配度 |
| 开发 → 审查 | `check-code-style` | 代码风格、规范合规 |
| 审查 → 测试 | `check-review-pass` | Review 通过、无遗留问题 |
| 测试 → 交付 | `check-test-coverage` | 覆盖率、功能通过率、性能达标 |

### 6.2 门禁判定逻辑

```
PASS    → 全部校验通过，进入下一环节
WARN    → 有警告项，标记后可推进（PM 决策）
FAIL    → 有硬性不通过项，必须回退修复
BLOCK   → 发现重大风险，暂停流水线，等待人工介入
```

## 7. MCP 外接体系

### 7.1 外接能力矩阵

| 系统 | 能力 | 数据流向 |
|------|------|---------|
| Git | 代码托管、PR 管理、分支策略 | 双向 |
| CI/CD | 自动构建、部署、流水线状态 | 双向 |
| 项目看板 | 任务状态、进度追踪、工时统计 | 双向 |
| 工单系统 | Bug 上报、需求登记、问题追踪 | 双向 |
| 监控系统 | 运行指标、告警通知、性能数据 | 输入 |
| 知识库 | 文档检索、规范查询、经验查询 | 输入 |

### 7.2 MCP 接入规范

每个 MCP 集成需定义：
- 接口协议（REST/gRPC/WebSocket）
- 数据格式（请求/响应 Schema）
- 鉴权方式
- 错误处理策略
- 降级方案

## 8. 资产沉淀体系

### 8.1 资产类型

| 资产 | 说明 | 更新时机 |
|------|------|---------|
| dev-map | 全域工程图谱，展示系统架构、模块依赖、技术栈 | 每次架构变更 |
| knowledge-base | 规范文档、技术选型、最佳实践 | 持续积累 |
| lesson-learned | 踩坑记录、故障复盘、优化经验 | 每次问题解决后 |
| templates | SPEC 模板、架构模板、测试模板 | 按需创建 |

### 8.2 沉淀机制

- 每个 Workflow 完成后，自动提取可沉淀资产
- 代码审查发现的共性问题，自动归入 lesson-learned
- 优化后的 Skill 流程，自动更新 Skill 库
- 架构变更后，自动更新 dev-map
