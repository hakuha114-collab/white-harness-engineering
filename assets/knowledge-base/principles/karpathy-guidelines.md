# Karpathy 编程四原则

> 来源：[andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 基于 Andrej Karpathy（前 Tesla AI 总监、OpenAI 联合创始人）对 LLM 编程的深度观察

## 三大致命问题

Andrej Karpathy 观察到 LLM 辅助编程时反复出现：

1. **错误假设且不验证** — 默默选择一种解释然后执行，不检查、不澄清
2. **过度复杂化** — 偏好臃肿抽象，100 行能搞定的事写出 1000 行
3. **附带破坏** — 顺手改不理解的代码，即使与任务无关

## 四大原则速查

### 1. 先思考再编码 (Think Before Coding)

- 明确陈述假设，不确定就问
- 多种理解时呈现所有选项
- 有更简单方案就说出来
- 不清楚就停下来请求澄清

### 2. 简洁优先 (Simplicity First)

- 不添加未被要求的功能
- 不为单次使用创建抽象
- 不添加未被请求的灵活性
- 如果 200 行可以写成 50 行，就重写

**检验**：高级工程师会说这过度复杂了吗？

### 3. 精准修改 (Surgical Changes)

- 不"改进"相邻代码
- 不重构没坏的东西
- 匹配现有风格
- 每行修改都能追溯到用户请求

### 4. 目标驱动执行 (Goal-Driven Execution)

- 定义成功标准
- 任务转化为可验证目标
- 多步骤任务每步可验证
- 循环直到验证通过

## 核心洞见

> **"过度复杂"的示例并非明显错误——它们遵循了设计模式和最佳实践。问题在于时机：在需要之前就添加了复杂度。**
>
> **好的代码是简单解决今天问题的代码，而不是过早地解决明天问题的代码。**

## 反模式 vs 正确做法

| 反模式 | 正确做法 |
|--------|---------|
| 默默假设文件格式、字段、范围 | 明确列出假设，请求澄清 |
| 用策略模式做单一折扣计算 | 只用一个函数，直到真正需要复杂度 |
| 修 Bug 时顺手改引号风格、加类型提示 | 只修改解决问题的那几行 |
| "我会审查并改进代码" | "写测试复现 Bug → 使其通过 → 验证无回归" |
