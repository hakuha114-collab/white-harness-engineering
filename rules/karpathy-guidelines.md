# Karpathy 编程原则红线

> 来源：[andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)
> 基于 Andrej Karpathy 对 LLM 编程三大致命问题的观察，提炼为四大约束原则

## 📌 背景问题

Andrej Karpathy 观察到 LLM 在辅助编程时反复出现三大致命问题：

1. **错误假设且不验证** — 模型默默选择一种解释然后执行，不检查、不澄清、不暴露不一致
2. **过度复杂化** — 偏好臃肿的抽象、过度工程，100 行能搞定的事写出 1000 行
3. **附带破坏** — 作为副作用更改/删除不理解的代码，即使与任务无关

## 🔴 四大约束原则

### 原则一：先思考再编码 (Think Before Coding)

> **不假设。不隐藏困惑。呈现权衡。**

- [ ] 实现前**必须**明确陈述假设，不确定就问，不许猜
- [ ] 存在多种理解时，**必须**呈现多种解读，不许默默选一种
- [ ] 如果有更简单的方案，**必须**说出来，适时反驳
- [ ] 如果不清楚，**必须**停下来，说出困惑，请求澄清

**违反等级**: L1（开发前必须执行思考步骤）

### 原则二：简洁优先 (Simplicity First)

> **用最少代码解决问题。不做推测性开发。**

- [ ] **禁止**添加未被要求的功能
- [ ] **禁止**为单次使用的代码创建抽象
- [ ] **禁止**添加未被请求的"灵活性"或"可配置性"
- [ ] **禁止**为不可能发生的场景写错误处理
- [ ] 如果 200 行可以写成 50 行，**必须**重写

**检验标准**：一个高级工程师会说这过度复杂了吗？如果是，就简化。

**违反等级**: L1（过度工程等同于违反代码规范）

### 原则三：精准修改 (Surgical Changes)

> **只触碰必须的。只清理自己造成的混乱。**

编辑现有代码时：

- [ ] **禁止**"改进"相邻的代码、注释或格式
- [ ] **禁止**重构没有坏的东西
- [ ] **必须**匹配现有风格，即使你不会那样写
- [ ] 发现无关死代码时**只提一下**，不删除

当你的修改产生孤立代码时：

- [ ] **必须**移除你的修改导致不再使用的 import/变量/函数
- [ ] **禁止**删除之前就存在的死代码（除非被要求）

**检验标准**：每一行修改都应该能直接追溯到用户的请求。

**违反等级**: L1（无关修改等同于代码审查不通过项）

### 原则四：目标驱动执行 (Goal-Driven Execution)

> **定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：

| ❌ 模糊表述 | ✅ 转化为 |
|------------|----------|
| "添加验证" | "为无效输入写测试，然后让测试通过" |
| "修复 bug" | "写一个能复现的测试，然后让测试通过" |
| "重构 X" | "确保重构前后测试都通过" |

多步骤任务**必须**声明计划：

```
1. [步骤] → 验证: [检查方式]
2. [步骤] → 验证: [检查方式]
3. [步骤] → 验证: [检查方式]
```

**违反等级**: L1（无成功标准的任务不允许推进）

## 🟢 权衡说明

> 这些原则偏向**谨慎而非速度**。对于简单任务（简单错别字修复、明显的一行修改），请自行判断——不是每个改动都需要完整的严格流程。
>
> 目标是减少非琐碎工作中的代价高昂的错误，而不是拖慢简单任务。
