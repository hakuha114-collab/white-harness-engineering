# Code Review 硬性检查项

> L1 级强制规则，审查 Agent 必须逐项核验

## 1. 功能正确性

- [ ] 代码实现与需求 SPEC 一致
- [ ] 边界条件已处理
- [ ] 异常情况已处理（网络超时、数据为空、并发冲突等）
- [ ] 无逻辑错误（条件判断、循环边界、类型转换等）
- [ ] 无资源泄露（连接未关闭、内存未释放等）

## 2. 代码质量

### 2.1 可读性

- [ ] 命名清晰表达意图
- [ ] 函数职责单一
- [ ] 嵌套层级不超过 3 层
- [ ] 无重复代码（DRY 原则）
- [ ] 复杂逻辑有注释说明

### 2.2 可维护性

- [ ] 遵循 SOLID 原则
- [ ] 模块间耦合度低
- [ ] 配置与代码分离
- [ ] 魔法数字/字符串已提取为常量
- [ ] 硬编码的路径/地址已提取为配置

### 2.3 性能

- [ ] 无 N+1 查询问题
- [ ] 无不必要的循环嵌套
- [ ] 大数据量操作有分页/流式处理
- [ ] 无同步阻塞操作在异步上下文中
- [ ] 缓存策略合理

## 3. 安全合规

### 3.1 数据安全（参照 security-rules.md）

- [ ] 无硬编码密钥/密码
- [ ] 无 SQL 注入风险
- [ ] 无 XSS 风险
- [ ] 敏感数据已加密/脱敏

### 3.2 访问控制

- [ ] 接口有认证和授权
- [ ] 权限校验完整
- [ ] 无越权访问风险

## 4. 测试覆盖

- [ ] 单元测试覆盖核心逻辑
- [ ] 测试用例覆盖正常路径和异常路径
- [ ] 边界值有测试
- [ ] 测试可独立运行，无外部依赖
- [ ] 测试命名清晰，表达测试意图

## 5. 工程规范

### 5.1 版本控制

- [ ] Commit 消息符合 Conventional Commits
- [ ] 无无关变更混入
- [ ] 无调试代码残留（console.log, debugger 等）
- [ ] 无注释掉的代码

### 5.2 文档

- [ ] 公共 API 有文档注释
- [ ] README 已更新（如涉及新功能）
- [ ] 变更日志已更新（如涉及 API 变更）
- [ ] 配置项有说明文档

## 6. 架构一致性

- [ ] 新增代码与现有架构风格一致
- [ ] 分层正确（不跨层调用）
- [ ] 依赖方向正确（不反向依赖）
- [ ] 新增模块位置合理
- [ ] 未引入循环依赖

## 7. Review 流程规范

### 7.1 Review 级别

| 级别 | 触发条件 | Review 要求 |
|------|---------|-------------|
| L1 轻量 | 文档修改/样式调整 | 1 人通过 |
| L2 标准 | 功能开发/Bug 修复 | 2 人通过 + 自动化检查通过 |
| L3 严格 | 核心模块/架构变更 | 3 人通过 + 技术负责人确认 + 自动化检查通过 |

### 7.2 Review 时效

- L1: 4 小时内完成
- L2: 24 小时内完成
- L3: 48 小时内完成

### 7.3 Review 产出

- Review 意见必须分级：`[MUST]` 必须修改 / `[SHOULD]` 建议修改 / `[NICE]` 可选优化
- 必须有明确的通过/不通过结论
- 不通过必须附具体修改建议
